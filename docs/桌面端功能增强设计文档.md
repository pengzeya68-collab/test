# TestMaster 桌面端功能增强实施规格

| 项目 | 内容 |
|---|---|
| 文档版本 | v1.1 |
| 编写日期 | 2026-07-21 |
| 文档状态 | 实施基线 |
| 适用产品 | TestMaster（Electron + Vue 3 + FastAPI + PostgreSQL） |
| 替代文档 | v1.0《桌面端功能增强设计文档》 |

## 1. 目标与边界

本规格的目标不是简单增加功能入口，而是让 TestMaster 成为可在企业环境稳定运行的自动化测试工作台：测试资产可追溯、执行可恢复、敏感数据受控、失败产物可复现、桌面端和服务器职责清晰。

首期范围：套件持久化、统一执行记录与产物、HAR 导入、受控 Mock 故障注入、浏览器内流量捕获、缺陷报告基础能力。

后续范围：失败归因、需求生成、定位器建议、桌面 Agent、外部代理抓包、第三方缺陷系统双向同步。

不在首期范围：安装根证书的 HTTPS 中间人代理、绕过验证码或反爬限制、无人工约束的 AI 自动修改正式用例、将真实生产敏感流量上传给 AI。

## 2. 真实现状与必须修正的前提

1. 自动化模型已接入主数据库，`core/autotest_database.py` 只保留兼容别名，不再维护独立 autotest 数据库。
2. `routers/autotest_suites.py` 当前通过 `autotest_data/suites.json` 保存“场景套件”；现有 `TestSuite` 模型没有被该路由使用，且其关联对象是接口用例而非场景。
3. `RecorderSession` 当前仅发送失败的网络事件摘要，不包含正常请求、请求体、响应体或请求头，不能直接转接口用例。
4. UI 自动化的截图、trace、未来视频均产生在桌面端本地文件系统。后端无法通过 IPC 反向读取本地文件。
5. 当前 Playwright 仅保证 bundled Chromium；跨浏览器支持不是本期承诺。

因此，任何新能力不得建立在“套件已经数据库化”“录制器已抓到完整流量”“后端能读取桌面端文件”这三个错误前提上。

## 3. 架构决策

### 3.1 执行职责

| 执行类型 | 权威服务 | 执行位置 | 说明 |
|---|---|---|---|
| 接口用例/场景/接口套件 | 后端任务服务 | 服务端 Worker | 支持调度、Webhook、CI |
| UI 用例/UI 套件手工运行 | 桌面端 | Electron + bundled Chromium | 使用本地登录态、录制与调试 |
| UI 无人值守运行 | 桌面 Agent（后续） | 受管 Windows 主机 | Agent 心跳、租约、任务领取 |
| 报告、历史、产物元数据 | 后端 | PostgreSQL + 产物存储 | 后端是唯一事实来源 |

桌面托盘仅保证应用未退出时提供本地体验；它不是可靠的服务端调度器。关闭应用、设备休眠、系统重启时，任务必须明确标记为 `waiting_for_agent`，不能伪造为已执行。

### 3.2 执行状态机

统一执行状态：

```text
queued -> assigned -> running -> passed
                            -> failed
                            -> cancelled
                            -> timed_out
                            -> infra_error
queued -> waiting_for_agent -> assigned
```

状态只允许沿上图迁移。每次迁移写入事件表，携带 `execution_id`、操作者/Agent、时间、错误码和幂等键。超时、取消和重试均创建新 attempt，不覆盖原结果。

### 3.3 产物边界

桌面端负责创建截图、trace、video、DOM 快照；后端负责签发上传、保存元数据、授权下载与生成报告。

```text
桌面端生成文件 -> SHA-256/大小/类型校验 -> 上传会话 -> 分片上传
-> 后端 ArtifactManifest -> 报告/下载/保留策略
```

禁止把 trace、视频或 HAR 转成 base64 放进普通 JSON 接口。上传必须支持大小限制、断点续传、哈希校验和失败重试。

### 3.4 安全与隐私基线

所有导入、抓包、AI、产物能力都遵守以下规则：

- 默认脱敏：`Authorization`、`Cookie`、`Set-Cookie`、`password`、`token`、`secret`、`api_key`、手机号、身份证等字段。
- 原始敏感值默认只保留在内存，不入库、不上传、不发送给 AI；用户显式批准后才可加密本地保存。
- 产物和导入文件具备所属用户、项目、保留期、下载权限和审计记录。
- 上传、HAR 导入、需求文档均设文件数量、单文件大小、解压后大小、解析深度和超时上限。
- 生产环境禁用 `DISABLE_SSL_VERIFY` 与 `DISABLE_SSRF_GUARD`。
- 外部 Webhook 使用随机令牌之外还必须校验 HMAC 签名、时间戳、重放窗口和幂等键。

## 4. Phase 0：平台基础改造

### 4.1 数据模型

新增或完成迁移的模型均使用主数据库并由 Alembic 管理，不使用 JSON 文件作为主存储。

| 模型 | 核心字段 | 作用 |
|---|---|---|
| `TestSuite` | `id,user_id,name,description,default_env_id,kind,is_active` | 套件实体 |
| `TestSuiteScenario` | `suite_id,scenario_id,sort_order` | 场景套件成员 |
| `TestSuiteSchedule` | `suite_id,cron,timezone,enabled,notification_config,next_run_at` | 服务器定时配置 |
| `AutomationExecution` | `id,type,target_id,status,attempt,idempotency_key,runner_id,started_at,finished_at` | 统一执行记录 |
| `AutomationExecutionItem` | `execution_id,target_type,target_id,status,sequence,result_summary` | 套件成员结果 |
| `ExecutionEvent` | `execution_id,sequence,level,event_type,payload_redacted` | 日志与状态流 |
| `ArtifactManifest` | `execution_id,kind,filename,size_bytes,sha256,storage_key,retention_until` | 产物元数据 |
| `CaptureSession` | `id,user_id,origin,started_at,ended_at,status,policy_version` | 流量捕获会话 |
| `CapturedExchange` | `session_id,sequence,request_redacted,response_redacted,fingerprint,selected` | 已脱敏流量 |
| `ImportJob` | `id,user_id,source_type,status,summary,error_summary` | 导入的预览、确认和审计 |

旧 JSON 套件迁移步骤：备份 -> 读取并校验 -> 按原 ID 映射创建数据库套件 -> 写入场景成员顺序 -> 比对数量/归属 -> 双读验证 -> 切换读路径 -> 保留只读备份。迁移必须可重复执行且不创建重复套件。

### 4.2 API 契约

所有新接口使用 `/api/auto-test` 前缀，响应统一返回资源或 `{data, meta}`，错误使用稳定 `code`。

| 接口 | 语义 |
|---|---|
| `POST /suites/{id}/executions` | 创建执行，返回 `202` 与 `execution_id` |
| `GET /executions/{id}` | 获取状态、摘要和成员结果 |
| `GET /executions/{id}/events` | 分页事件流；桌面端可轮询或 SSE |
| `POST /executions/{id}/cancel` | 请求取消，幂等 |
| `POST /artifacts/upload-sessions` | 创建受限上传会话 |
| `POST /artifacts/upload-sessions/{id}/parts` | 分片上传 |
| `POST /artifacts/upload-sessions/{id}/complete` | 校验哈希并创建 manifest |

Webhook 创建执行后立即返回 `202`，不在 HTTP 请求内等待完整套件运行。调用方通过 `execution_id` 查询或接收签名回调。

### 4.3 调度与可靠性

服务端调度使用 PostgreSQL 作为持久化来源；多实例必须由数据库/Redis 租约保证同一 schedule 同一时刻只领取一次。每个任务包含时区、misfire 策略、并发数、超时、重试上限和幂等键。

桌面/Agent 领取 UI 任务必须 30 秒心跳续租；心跳超时后任务回到 `waiting_for_agent` 或 `infra_error`，不得继续显示为 `running`。

## 5. F3：统一导入中心与 HAR

### 5.1 支持范围

首期支持 Postman Collection v2.1、OpenAPI 3.x、Swagger 2.0、HAR 1.2、cURL 文本、已验证结构的 Apipost 导出。文件类型由内容结构和用户确认共同决定，扩展名只作提示。

### 5.2 处理流程

```text
选择/拖拽文件 -> 安全解析和大小校验 -> 标准化 ImportCandidate
-> 预览/脱敏/冲突检测 -> 用户确认策略 -> 单事务导入 -> ImportJob 结果
```

`ImportCandidate` 必须保留来源、原始名称、HTTP 方法、规范化 URL、请求配置、默认断言、敏感字段标记和稳定指纹。冲突不能只依赖名称；使用 `method + canonical_url + canonical_request_shape` 指纹，并让用户选择跳过、创建副本或显式更新。

### 5.3 HAR 规则

- 仅解析 HTTP(S) entry，不承诺将 WebSocket 自动变成接口用例。
- 支持 JSON、form-urlencoded、multipart 元数据和 base64 内容摘要；二进制文件不写入普通用例 body。
- 处理重定向链时保留最终请求，原始链路仅留在导入审计。
- 请求和响应的最大可保存正文为 1 MB；超过上限仅保存摘要和大小。
- `Cookie`、鉴权头和敏感参数被脱敏，恢复原值只能通过环境密钥引用，不从 HAR 复制到用例。

## 6. F6：受控 Mock 故障注入

首期支持应用层故障：`status_error`、`delay`、`timeout_response`、`invalid_json`、`custom_headers`。规则保存 `fault_type`、`fault_config`、`trigger_probability`、`random_seed` 和启用状态。

`connection_reset`、`partial_body` 属于传输层故障，不在 FastAPI JSONResponse 层伪实现；后续通过专用代理或 StreamingResponse 的受限实现提供，并在 UI 中明确标识支持范围。

概率规则需支持固定 seed 复现，响应日志必须记录“规则是否命中、随机值、故障类型”，否则无法稳定回归。

## 7. F1：浏览器内流量捕获

### 7.1 采集范围

首期仅采集由 TestMaster 启动的 Chromium 页面发出的 XHR/fetch 请求。忽略图片、字体、媒体、样式、脚本和导航文档；用户可配置域名白名单、路径排除和采集暂停。

每个 exchange 采集：请求方法、规范化 URL、已脱敏请求头、受限请求体、响应状态、已脱敏受限响应体、耗时、资源类型、失败原因和关联页面 URL。

### 7.2 请求转用例与转场景

捕获结果先生成候选，不直接入库。用户选择请求后进入参数化页面：

- 选择稳定字段和动态字段。
- 对响应 JSONPath 指定变量提取规则。
- 在后续请求中将匹配到的动态值替换为 `{{variable}}`。
- 为每个请求选择状态码、JSONPath、schema 或业务字段断言。
- 生成场景前显示依赖图；无法推导的依赖必须由用户确认。

只有经预览运行成功的候选场景才能标记为“可执行”。“登录到下单自动通过”不是默认承诺，而是该具体流程在用户确认变量映射后通过的验收条件。

### 7.3 验收

- 正常 XHR/fetch 可捕获，非 API 静态资源不会进入候选列表。
- 鉴权头、Cookie、密码不会出现在界面、日志、数据库或 AI 请求中。
- 通过变量映射将登录返回的 token、下单返回的订单 ID 传递给后续请求。
- 捕获、导入、取消和超限都有可理解错误提示与审计记录。

## 8. F5：缺陷报告与产物

Playwright video 在 Context 关闭后才有稳定文件路径。执行器必须先关闭并固化 trace/video，再创建上传任务和报告，不允许在文件未完成时生成报告。

缺陷报告由后端按 ArtifactManifest 生成，包含：执行版本、环境脱敏摘要、失败步骤、断言差异、截图、trace/video 下载链接、重现前提和时间线。报告不嵌入大型二进制内容。

截图标注保存为独立 JSON 标注层，原截图不可变；导出时再渲染，确保审计与重编辑能力。

## 9. F2：AI 增强的受控模式

### 9.1 失败归因

输出是带置信度的建议，不是事实判定。结构固定为：`category`、`confidence`、`evidence`、`unknowns`、`next_actions`。类别为环境、数据、产品缺陷、脚本、未知。

验收以人工标注失败集衡量：证据完整率、Top-3 建议命中率、人工纠正率、耗时与单次成本。禁止使用“能够正确区分”这类不可测表述。

### 9.2 定位器建议

默认仅建议和暂停，永不直接修改正式用例。候选必须经过：唯一匹配、可见性、可操作性、当前 URL/登录态一致性和 dry-run 校验。用户确认后才创建新版本；自动应用仅允许在测试分支、白名单项目和明确阈值下启用。

页面 DOM、可访问性快照和截图在发送 AI 前必须裁剪、脱敏，并防御页面文本中的提示注入。

### 9.3 需求生成

需求生成需先生成可编辑的测试点与需求追踪编号，再生成用例。每个生成用例必须显示来源测试点、假设和缺失信息；不允许把模型猜测伪装为已验证业务规则。

## 10. 权限、观测与验收

新增权限：`suite:schedule`、`execution:read`、`execution:cancel`、`artifact:download`、`capture:create`、`capture:export`、`ai:analyze`、`mock:fault-inject`。默认最小权限，敏感原始流量与产物下载分离授权。

每项功能必须具备：结构化日志、错误码、审计事件、性能指标、健康检查、单元测试、接口测试、桌面端集成测试和失败注入测试。

发布前验收矩阵至少覆盖：多用户隔离、服务重启、桌面退出、网络中断、断点上传、重复 webhook、任务取消、超时、敏感数据扫描、迁移回滚和 PostgreSQL 多实例。

## 11. 实施顺序与工作量

| 阶段 | 交付 | 估计 |
|---|---|---|
| Phase 0 | 套件数据库迁移、统一执行/事件/产物、调度租约 | 3-5 周 |
| Phase 1 | HAR/统一导入、受控 Mock 应用层故障 | 2-3 周 |
| Phase 2 | 浏览器流量捕获、变量关联、产物上传、缺陷报告 | 4-6 周 |
| Phase 3 | 失败归因、需求生成 | 3-4 周 |
| Phase 4 | 定位器建议、桌面 Agent、传输层故障、外部代理评估 | 6-10 周 |

Phase 0 未完成前，不实施套件级无人值守、趋势看板或 UI 无人值守。每阶段必须通过发布前验收矩阵，才可进入下一阶段。

## 12. 本轮实施承诺

本轮代码实施顺序严格遵守本规格：先移除套件 JSON 主存储和补齐执行/产物基础能力；再实现 HAR 与 Mock；最后实施桌面流量捕获。任何需要桌面/后端跨进程传文件的能力都通过 ArtifactManifest 上传协议实现，不引入后端读取 IPC 文件的反向依赖。
