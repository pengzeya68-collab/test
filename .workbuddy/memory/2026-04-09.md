# 2026-04-09 工作日志

## BUG修复：执行报告详情步骤数显示错误

### 问题描述
执行历史详情弹窗显示错误步骤数（显示10个，实际只有2个）

### 根本原因（三层问题叠加）
1. **前端**：axios响应拦截器配置不一致，`autoTestRequest`缺少统一的数据解套
2. **后端**：`get_report_detail` API使用过期的`report.total_count`字段而非实时计算
3. **数据库**：存在脏数据——同一report_id下累积了多次执行的orphan步骤记录

### 修复方案
1. **frontend/src/utils/request.js**：添加统一的响应拦截器
2. **backend/api/auto_test.py**：`get_report_detail`改为从`formatted_steps`实时计算数量
3. **数据库清理**：删除orphan记录，只保留最新执行的步骤数据

### 预防措施建议
- API返回数据应始终基于实际数据实时计算
- 执行结果入库前应先清理同一report_id的旧数据
- 添加数据库事务确保原子性

## 追加修复：执行结果返回结构不匹配

### 问题描述
执行完成后前端显示：总步骤0，成功0，失败0，且报告链接无法拼接

### 根本原因
后端 `get_task_status` API 返回结构是嵌套的 `{task_id, status, data: {step_results, report_id}}`
但前端直接访问 `res.step_results`、`res.report_id`，取不到值

### 修复方案
**backend/api/auto_test.py**：`get_task_status` 改用 `response.update(data)` 将 celery 返回数据合并到顶层

- backend/api/auto_test.py
