### 后端核心功能

#### 1. 场景执行引擎 (scenario_runner.py)
- **ScenarioExecutionEngine**：按顺序执行场景中的每个步骤
- **全局上下文传递**：通过 context_vars 存储提取的变量（如 token），后续步骤自动使用
- **变量替换**：URL、Headers、Payload 支持 `{{variable_name}}` 占位符
- **has_failed 标志位**：失败后继续处理后续步骤（标记为 skipped）
- **默认兜底断言**：无 status_code 断言时自动检查 2xx/3xx
- **DataDrivenScenarioExecutionEngine**：数据驱动执行，支持多行数据迭代

#### 2. 定时任务调度器 (scheduler.py)
- 使用 APScheduler AsyncIOScheduler
- 支持 Cron 表达式定时执行
- API 端点：创建/删除/查询/手动触发定时任务

#### 3. 单接口执行 (execution.py)
- quick_run_case：快速执行（不保存历史）
- run_case_with_pytest：使用 Pytest 执行并生成 Allure 报告
- 断言规则解析：支持新旧两种格式

## 今天修复的 Bug

### Bug 1: 场景步骤丢失
- **问题**：6步场景第3步失败时，4/5/6步丢失，前端只收到3步
- **原因**：`scenario_runner.py` 使用 `break` 导致后续步骤未处理
- **修复**：改用 `has_failed` 标志位 + `continue`，确保所有步骤都添加到结果

### Bug 2: 断言逻辑兜底
- **问题**：接口返回 422/500 等错误时仍显示"通过"
- **修复**：无断言时默认检查 2xx/3xx，无 status_code 断言时也添加兜底检查

- **问题**：没有指定 env_id 时无法获取环境变量

- **问题**：相对路径 URL 没有拼接 base_url
- **修复**：自动拼接 base_url

## 严重 Bug 修复：场景执行步骤丢失

### 问题描述
当场景有 6 个步骤，第 3 步失败时，前端只收到 3 个步骤数据，4/5/6 步丢失！

### 根本原因
`scenario_runner.py` 中使用 `break` 语句导致失败后的步骤未被添加到 `step_results`

### 修复内容
`auto_test_platform/services/scenario_runner.py`

使用 `has_failed` 标志位替代 `break`，确保所有步骤都被处理：
- 前面有步骤失败时，后续步骤标记为 `skipped` 状态，`skipped_reason` 说明原因
- 继续执行循环直到所有步骤都处理完

### 验证结果
API 直接调用返回完整的 6 个步骤数据（包含 3 个 skipped）
